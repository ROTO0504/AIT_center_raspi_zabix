1. システム概要

zabix-alarm は、Zabbix が送信するアラートメールを Microsoft 365 メールボックス上で監視し、その内容に応じて Raspberry Pi に 接続された LED とブザーで物理的な障害通知を行うシステムである。 Web ダッシュボードからリアルタイムに障害状況・ホスト一覧・操作ログを 閲覧でき、ボタン押下で通知音を一時停止することもできる。

1.1 設計の前提

  • 監視対象のアラートは Zabbix から特定メールボックス宛に届く
  • メールアクセスは Microsoft Graph API（MSAL によるアプリ認証）
  • 判定対象は high・disaster 相当の重大度のみ通知に反映
  • 復旧（Resolved）メールはステータス更新のみで音は鳴らさない

1.2 主な機能

  • メールフォルダの定期ポーリングと判定
  • 赤 LED 点灯 + ブザー 2 回（重大障害検出時）
  • 緑 LED 点滅（起動時の初回判定待機表示）
  • ボタン押下で LED 消灯 + ブザー 1 回（音だけ静かにしたいとき用）
  • Web ダッシュボードでホスト別状態 / ログ / 動作中バージョンを表示
  • Gitea Actions による Mac → ラズパイの自動デプロイ

1.3 通知レベル

high / disaster
赤 LED 点灯。ブザー 2 回。Web は赤色表示。
normal（OK / 復旧）
通知なし、状態のみ更新。Web は緑色表示。
unknown
未判定 / 該当メール未受信。Web は灰色表示。

3. ハードウェア仕様 / GPIO ピン配置

3.1 機材

  • Raspberry Pi 4 / 5（aarch64 / arm64）
  • 赤色 LED ×1（GPIO 26）
  • ブザー ×1（GPIO 5）
  • 押しボタン ×1（GPIO 12、内部プルアップ）
  • 抵抗・配線材一式
GPIO 6（緑）と GPIO 13（黄）はコード上に定義は残るが、現運用では赤・ブザー・ボタンのみ使用。 緑 LED 点滅は「起動時の初回判定待機」の状態フラグとして残されている。

5. 動作仕様

5.1 起動シーケンス

  1. systemd が start_system.py --skip-install を起動
  2. Flask Web サーバ（0.0.0.0:8080）と監視スレッドが立ち上がる
  3. 初回メール判定が完了するまで 緑 LED が点滅（起動中表示）
  4. 判定が確定したら点滅停止、結果に応じた状態に遷移

5.2 通常ループ

  • 設定された MAIL_POLL_INTERVAL（既定値）ごとに最新メールを取得
  • 件名・本文から重大度（high / disaster / average / 復旧）を抽出
  • ホスト単位の状態テーブルを更新し、全体の overall を計算
  • 状態が前回と変わっていれば LED とブザーを操作、ログに記録
  • 判定に該当しないメールは WARN ログに記録（取りこぼし監視のため）

5.3 ボタン仕様

  • GPIO 12 を pull-up で入力。押下で GND に落ちる
  • 押下 → LED と赤・ブザーを消灯（silenced 状態）
  • silenced 中に新しい重大障害メールが入れば、改めて点灯・鳴動
  • ステータス表示（overall）はボタンでは変えない

5.4 復旧時の挙動

過去の重大障害が解消された場合、ホスト状態は ok に戻るが、 音や LED は鳴らさない。これは「復旧通知が連続するメール群で 繰り返し鳴る」のを避けるためで、運用中の意図的な仕様である。

6.4 Gitea Actions runner（act_runner）

# バイナリ取得（aarch64 / Pi 4・5）
cd ~
wget https://gitea.com/gitea/act_runner/releases/download/v0.5.0/act_runner-0.5.0-linux-arm64 -O act_runner
chmod +x act_runner
sudo mv act_runner /usr/local/bin/

# 登録（ホスト実行モード）
mkdir -p ~/act_runner && cd ~/act_runner
act_runner register \
  --no-interactive \
  --instance http://192.168.1.201:3000 \
  --token <Gitea で発行した runner token> \
  --name raspi \
  --labels "raspi:host"

# systemd 常駐化
sudo tee /etc/systemd/system/act_runner.service >/dev/null <<'EOF'
[Unit]
Description=Gitea act_runner
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=raspi
WorkingDirectory=/home/raspi/act_runner
ExecStart=/usr/local/bin/act_runner daemon
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target
EOF

sudo systemctl daemon-reload
sudo systemctl enable --now act_runner
Gitea 管理画面 → Actions → Runners に raspi (Online) が出れば成功。

7.5 ラズパイ側の git の使い方（重要）

ラズパイ側で直接コードを編集しないこと。 自動デプロイのジョブが git reset --hard AIT-center-mac/main を実行するため、 ラズパイ上での未コミットの変更や独自コミットは次回デプロイで全て破棄される。 必ず Mac 側でコミット → push する運用を守る。

7.8 認証情報の取り扱い

  • Gitea のパスワードは git config credential.helper store でラズパイにのみ保存（平文 ~/.git-credentials。ローカルネットワーク前提のため許容）
  • Mac 側は macOS Keychain に保存される（credential.helper=osxkeychain がデフォルト）
  • Azure 認証情報は .env のみに置く。コミットや Slack 等への貼り付け禁止

7.9 トラブル時の安全な戻し方

本番でおかしくなった場合、リバートコミットを作って push するのが基本。 ラズパイ上で直接 git reset --hard <古いハッシュ> しても、 次回 push で再上書きされるので根本解決にならない。 

# Mac で

git push AIT-center-mac main
# → Actions が走り、ラズパイがリバート版に更新される

10. トラブルシューティング

10.1 サービスの状態確認

sudo systemctl status mail2alertlight
sudo journalctl -u mail2alertlight -n 50 --no-pager
sudo journalctl -u mail2alertlight -f       # ライブ追従

10.2 動作中バージョンが unknown と出る

systemd は root で動くため、raspi 所有の git ディレクトリで git rev-parse を実行すると "dubious ownership" で失敗するケースがある。 現バージョンでは web_app.py 内で -c safe.directory=... を渡して回避済み。それでも unknown が出る場合は .git の有無、 git バイナリの PATH、サービスの WorkingDirectory を確認する。

10.3 Web に接続できない

  • サービスが落ちている → systemctl status 確認
  • ポート競合 → WEB_PORT を別の値で再起動
  • 同じ LAN にいない → ラズパイの IP（既定 192.168.1.109）で疎通確認

10.4 Actions が走らない / runner が Offline

  • sudo systemctl status act_runner で daemon の生死確認
  • Gitea の管理画面 → Actions → Runners で再登録（古いトークンを削除）
  • Gitea 自体の Actions 機能が無効化されていないか確認

10.5 メールが取れない / 認証エラー

  • CLIENT_SECRET の有効期限切れ（2026/11/10 期限）
  • Azure 側で Mail.Read 権限の管理者同意が外れていないか
  • テナント ID / アプリケーション ID の打ち間違い
  • msal_token_cache.bin を削除して再取得を試す

10.6 LED やブザーが反応しない

  • 配線・電源確認（特に GND の共通化）
  • python3 gpio_test.py test-all で単体動作を確認
  • monitor_service.py の判定ログに該当アラートが流れているか
